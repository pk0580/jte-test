# План рефакторинга и оптимизации (REFACORING.md)

В данном документе зафиксированы выявленные проблемы архитектуры, производительности и чистоты кода, а также пошаговый план по их исправлению на основе проведенного аудита.

---

## 🔍 Выявленные проблемы

### 1. Нарушение SRP в Репозиториях
*   **Проблема:** `ArticleRepository`, `PayTypeRepository` и `OrderRepository` напрямую реализуют логику кэширования внутри методов.
*   **Влияние:** Затрудняет тестирование, нарушает разделение слоев (инфраструктура кэша в логике получения данных).

### 2. "Толстые" Use Cases
*   **Проблема:** `CreateOrderUseCase` напрямую оркестрирует низкоуровневые проверки существования сущностей в БД.
*   **Влияние:** Избыточная связанность. Изменения в логике валидации зависимостей требуют правки всех Use Cases.

### 3. Дублирование логики отказоустойчивости в поиске
*   **Проблема:** `ManticoreOrderSearch` и `CombinedSearchProvider` одновременно пытаются реализовать Fallback-стратегии.
*   **Влияние:** Размытая ответственность и сложность отладки механизмов Circuit Breaker.

### 4. Риски атомарности Outbox
*   **Проблема:** Сохранение событий Outbox через события Doctrine вне основной транзакции Use Case (в некоторых сценариях).
*   **Влияние:** Риск рассинхронизации данных между MySQL и поисковым индексом при сбоях.

### 5. Смешивание Read и Write моделей в статистике
*   **Проблема:** Тяжелые агрегационные запросы `getStats` находятся в `OrderRepository`.
*   **Влияние:** Снижение производительности репозитория и сложность масштабирования чтения статистики.

---

## 📋 План по исправлению

### Этап 1: Чистота кода и Архитектура
1.  **Repository Decorators:** Вынести кэширование из `ArticleRepository` и `PayTypeRepository` в отдельные декораторы.
2.  **UseCase Refactoring:** Перенести логику сборки заказа (hydration) из `CreateOrderUseCase` в `OrderFactory` или доменный сервис.
3.  **Outbox Improvements:** Добавить в `OutboxEvent` поле `scheduled_at` для управления очередью повторов.

### Этап 2: Производительность и High Load
1.  **Search Unification:** Убрать внутренний fallback из `ManticoreOrderSearch`, централизовать логику в `CombinedSearchProvider`.
2.  **Read-Model Separation:** Создать `OrderStatsProviderInterface` для получения статистики, отделив чтение от записи.
3.  **Batch Processing:** Оптимизировать `ProcessOutboxCommand` для пакетной отправки сообщений в шину.

### Этап 3: Надежность и Мониторинг
1.  **Transactional Integrity:** Гарантировать сохранение Outbox-событий в рамках основной транзакции бизнес-логики.
2.  **Contextual Logging:** Внедрить сквозное логирование `request_id` для отслеживания цепочки вызовов от API до фоновых команд.
3.  **Circuit Breaker Tuning:** Настроить параметры Circuit Breaker на основе профиля нагрузки внешних зависимостей.
