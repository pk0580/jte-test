# План рефакторинга и оптимизации (REFACORING.md)

В данном документе зафиксированы выявленные проблемы архитектуры, производительности и чистоты кода, а также пошаговый план по их исправлению.

---

## 🔍 Выявленные проблемы

### 1. Производительность Поиска (N+1 и избыточная гидрация)
*   **Проблема:** В `ManticoreOrderSearch::search` сначала получается список ID из Manticore, а затем `OrderRepository::findByIds` всегда гидрирует полные сущности `Order` и `OrderArticle` из MySQL.
*   **Влияние:** При больших результатах поиска (page/limit) нагрузка на БД неоправданно растет. В самом индексе Manticore уже есть часть данных (`number`, `email`, `client_name`), но они игнорируются для отображения в листингах.

### 2. Низкая отказоустойчивость API цен (`GetPrice`)
*   **Проблема:** `PriceParser` делает прямой синхронный HTTP-запрос к внешнему ресурсу (`tile.expert`) при каждом вызове API без кэширования.
*   **Влияние:** 
    - Отсутствие **кэширования** приводит к замедлению ответов и риску блокировки со стороны внешнего сервиса.
    - Отсутствие **Circuit Breaker** (предохранителя) означает, что при падении внешнего ресурса, PHP-воркеры (FPM) будут блокироваться до тайм-аута, что может привести к каскадному отказу всего API.

### 3. Архитектурные нарушения (Clean Architecture & SOAP)
*   **Проблема:** 
    - В `SoapController` путь к WSDL завязан на файловую систему проекта напрямую.
    - В `SoapOrderService` (Application слой) используется `Normalizer` напрямую для сборки массивов под SOAP — это детали реализации инфраструктуры, которые просочились в бизнес-логику.
    - **Дублирование валидации:** В `CreateOrderUseCase` валидация DTO вызывается вручную, хотя входные данные уже должны быть валидированы на уровне интерфейса (SOAP/REST).

### 4. Надежность Outbox
*   **Проблема:** `ProcessOutboxCommand` не имеет механизма повторных попыток (retry strategy). Если внешняя система (Messenger/Manticore) временно недоступна, событие помечается как обработанное или вызывает ошибку всей команды.

---

## 📋 План по исправлению

### Этап 1: Оптимизация производительности (High Load)
1.  **Search DTO:** Изменить `OrderSearchInterface` для возврата облегченных `SearchOrderDto`. Данные для простых листингов должны браться напрямую из Manticore.
2.  **Caching Decorators:** Вынести логику кэширования из `ArticleRepository` и `PayTypeRepository` в отдельные декораторы, разделяя ответственность.
3.  **Price Cache & Circuit Breaker:** Внедрить декоратор для `PriceParser` с использованием `Symfony Cache` (TTL ~1 час) и `Circuit Breaker`.

### Этап 2: Улучшение структуры (Clean Code)
1.  **Refactor SOAP Layer:**
    - Создать `WsdlProviderInterface` для абстракции получения WSDL.
    - Вынести логику конвертации SOAP-объектов в инфраструктурный уровень.
    - Убрать ручную валидацию из `UseCase`, полагаясь на автоматическую валидацию DTO.
2.  **Outbox Resilience:**
    - Добавить поля `attempts` и `last_error` в сущность `OutboxEvent`.
    - Реализовать в `ProcessOutboxCommand` стратегию повторов (Exponential Backoff).

### Этап 3: Инфраструктурные настройки
1.  **DB Indexes:** Оптимизировать индексы для таблицы `outbox_events` и внешних ключей `orders`.
2.  **Docker Limits:** Настроить лимиты ресурсов для контейнеров в `docker-compose.yml` (PHP-FPM, Manticore) для стабильной работы под нагрузкой.
