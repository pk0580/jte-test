### Полнотекстовый поиск в приложении

В приложении реализован высокопроизводительный полнотекстовый поиск на базе **Manticore Search**. Система спроектирована для работы с большими объемами данных (10M+ записей) с обеспечением отказоустойчивости и непрерывной доступности.

---

### 1. Архитектура поиска

Поиск разделен на несколько уровней согласно принципам **Clean Architecture**:

1.  **Domain (Слой бизнес-логики):**
    *   `OrderSearchInterface` — определяет контракт для поиска, индексации и мониторинга.
    *   `SearchResult<T>` — универсальный DTO для возврата результатов и общего количества (total count).
2.  **Infrastructure (Слой реализации):**
    *   `ManticoreOrderSearch` — основная реализация через клиент Manticore.
    *   `OrderSearchQueryBuilder` — выделенный сервис для построения сложных запросов, инкапсулирующий логику весов и пагинации.
    *   `SearchQueryDto` — стандартизированный объект параметров запроса.
3.  **Application (Слой сценариев):**
    *   `SearchOrdersUseCase` — координирует поиск и преобразование сущностей в DTO ответа API.

---

### 2. Функциональные возможности

#### Взвешенный поиск (Weighted Search)
Для повышения релевантности реализовано автоматическое добавление весов к полям:
*   `number ^5` (номер заказа — наивысший приоритет)
*   `email ^3`
*   `client_name ^2`, `client_surname ^2`
*   `company_name ^1`
*   `description` (базовый вес)

*Если пользователь использует в запросе спецсимволы Manticore (`@`, `|`, `^`), автоматическое взвешивание отключается для поддержки "экспертного" режима.*

#### Фильтрация и Типы данных
*   **Атрибуты:** Поле `status` проиндексировано как `int`, что позволяет выполнять быструю фильтрацию `WHERE status = ?`.
*   **Большие ID:** Индекс использует `id='bigint'`, что гарантирует совместимость с 64-битными идентификаторами из основной БД.
*   **Infix Search:** Настроено `min_infix_len='3'` для поиска по подстрокам (например, часть номера или email).

### 3. Производительность и Оптимизации

#### Оптимизация Bulk-индексации (Chunking)
Для минимизации сетевых задержек и нагрузки на CPU/RAM при массовой загрузке данных:
*   **Пакетная обработка (Chunking):** Данные разбиваются на порции по **1000 документов**.
*   **Контроль размера (Byte-size Limit):** Реализована проверка размера JSON-тела запроса. Если батч превышает **90% от 4MB** (константа `MAX_SQL_BYTES`), он отправляется немедленно, не дожидаясь наполнения по количеству.
*   Это предотвращает "memory spikes" на стороне PHP и ошибки `max_allowed_packet` на стороне Manticore.

#### Курсорная пагинация (Cursor-based Pagination)
Для предотвращения деградации производительности на "глубоких" страницах (Deep Paging):
*   Реализован механизм `last_id`. Если передан курсор, система игнорирует `offset` и использует фильтрацию по ID: `WHERE id < :last_id ORDER BY id DESC LIMIT 50`.
*   Это позволяет Manticore мгновенно находить следующую порцию данных, не сканируя и не сортируя миллионы предыдущих строк.
*   **Разделение режимов:** `OrderSearchQueryBuilder` строго разделяет обычную пагинацию (по страницам) и курсорную, исключая конфликты параметров.

#### Эффективная Гидрация (Batch Hydration)
Чтобы избежать проблемы **N+1** при получении данных из основной БД:
1.  Manticore возвращает только список ID найденных заказов.
2.  Система выполняет **один запрос** в MySQL через `WHERE id IN (...)` для получения полных объектов сущностей.
3.  Результаты сортируются в PHP согласно порядку, возвращенному поисковым движком (релевантность или кастомная сортировка).

#### Асинхронная индексация (Background Indexing)
Индексация вынесена из основного HTTP-потока с помощью **Symfony Messenger**:
*   При изменениях в БД (события Doctrine) в очередь `async` отправляются легковесные сообщения (`IndexOrderMessage`).
*   Worker обрабатывает их в фоновом режиме. Это снижает Latency API: пользователю не нужно ждать, пока завершится сетевой запрос к Manticore.

#### Схема данных и типы
*   **Attributes vs Text:** Поле `status` проиндексировано как `int` (атрибут), что позволяет выполнять мгновенную фильтрацию без полнотекстового разбора.
*   **Bigint IDs:** Использование `id='bigint'` предотвращает проблемы переполнения при больших объемах данных.
*   **Infix Indexing:** Оптимальный `min_infix_len='3'` обеспечивает баланс между скоростью поиска по частям строк (номера, email) и размером индекса на диске.

### 4. Отказоустойчивость и Zero-downtime

#### Атомарное переключение индексов (Atomic Swap)
Команда переиндексации `app:index-orders` использует механизм безопасной подмены:
1.  Создается временная таблица (например, `orders_tmp`).
2.  Выполняется **Zero-downtime Swap**:
    *   `orders` -> `orders_old` (переименование текущего).
    *   `orders_tmp` -> `orders` (активация нового).
3.  **Механизм отката (Rollback):** Если активация нового индекса не удалась, система автоматически пытается восстановить `orders_old` обратно в `orders`, логируя критическую ошибку при неудаче.

#### Fallback на БД (Resilience)
В случае падения Manticore Search система автоматически переключается на поиск через MySQL (`LIKE`). Это гарантирует доступность функционала поиска даже при аварии инфраструктуры.

#### Health-check & Monitoring
Реализован мониторинг состояния через `ping()` (выполняет `SELECT 1`), что позволяет балансировщикам или Kubernetes отключать трафик на неисправную ноду.

---

### 5. Команда индексации `app:index-orders`

Спроектирована для **Zero-downtime** обновления индекса:
1.  **Main Pass:** Копирует существующие данные во временный индекс.
2.  **Catch-up Pass:** Докачивает записи, созданные во время работы команды.
3.  **Atomic Swap:** Мгновенно подменяет основной индекс временным через `ALTER TABLE RENAME`.

**Параметры:**
*   `--batch`: Размер пачки (до 10000).
*   `--resume-from-id`: Продолжение работы с прерванного места.
